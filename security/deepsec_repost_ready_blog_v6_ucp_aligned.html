Develop Database-Enforced End-User Authorization with Oracle AI Database Deep Data Security, Spring Boot, UCP, and Microsoft Entra ID

This walkthrough shows how Microsoft Entra identity can flow through a Spring Boot application and a pooled JDBC connection so Oracle AI Database enforces end-user row, column, and cell-level authorization.

Feature image alt text: Diagram showing a browser user, Microsoft Entra tokens, a Spring Boot application, an Oracle UCP connection pool, Oracle JDBC Deep Data Security context propagation, and Oracle AI Database enforcing fine-grained access policies.

Key Takeaways

  • Pooled JDBC connections can still carry end-user identity when the application or JDBC provider attaches an Oracle JDBC end-user security context before SQL runs.
  • The OAuth on-behalf-of flow produces a separate database-access token, so the user token proves the browser user while the database token proves access to Oracle AI Database.
  • Microsoft Entra app roles such as EMPLOYEES and MANAGERS map to Oracle Deep Data Security data roles.
  • The same SQL and the same endpoint can return different rows and columns because Oracle AI Database enforces declarative data grants at execution time.
  • The repo now includes two aligned demos: an explicit JDBC API version and an ojdbc-provider-spring provider/SPI version. Both use Oracle UCP instead of HikariCP.

Why This Matters

Most enterprise Java services use a connection pool. That is good engineering: fast, stable, observable, and easy to operate. It also means many requests may reach the database through the same application database account.

That is where authorization gets tricky. If Emma and Marvin both use the same pooled database user, the database cannot naturally tell which rows belong to Emma, which columns Marvin may see, or which values a manager may update unless the application propagates end-user context.

Oracle Deep Data Security changes the shape of the problem. The application still uses a pooled connection, but an end-user security context is attached to the JDBC connection before SQL runs. Oracle AI Database evaluates declarative data grants at execution time.

This matters even more for AI and agentic applications. Agents, copilots, and natural-language SQL tools can fan out into many possible data paths. With Deep Data Security, the database remains the final policy enforcement point for fine-grained row, column, and cell-level access.

What We Are Building

The demo has two implementations of the same idea.

Version What it demonstrates Public endpoint JDBC pool
API version Application code obtains the signed-in user's token, obtains a database-access token, creates EndUserSecurityContext, calls OracleConnection.setEndUserSecurityContext(...), runs SQL, then clears the context. GET /deepsec/query Oracle UCP
Provider/SPI version Spring Security validates the OAuth 2.0 access token. ojdbc-provider-spring reads the authenticated Spring Security context through JDBC provider SPI and supplies the end-user security context to Oracle JDBC. GET /deepsec/query Oracle UCP

The entrance points are intentionally aligned. A user or reviewer can call GET /deepsec/query in either app and compare the internals instead of comparing different URLs. The difference is the token plumbing.

The Runtime Flow

  1. The client calls GET /deepsec/query.
  2. Spring Security authenticates the user or validates the incoming bearer token.
  3. The application obtains or exposes an end-user access token.
  4. A database-access token is obtained for the Oracle AI Database protected resource.
  5. Oracle JDBC receives an EndUserSecurityContext.
  6. The app can run session initialization SQL such as alter session disable parallel query before the business query.
  7. Oracle AI Database validates the context, activates mapped data roles from the token claims, and enforces data grants while SQL runs.
  8. The application returns the query result without hand-coding row or column authorization logic.

The Two Token Model

Token What it proves How the app uses it
End-user access token The signed-in browser or API caller is an authenticated Microsoft Entra user. The API version reads it from OAuth2AuthorizedClient. The provider version lets Spring Security validate it and makes it available through the request security context.
Database-access token The request is addressed to the Oracle AI Database protected resource. The API version requests it with on-behalf-of. The provider version requests it through the configured Spring OAuth2 client registration.

Aligned Controller Entry Points

The two demos intentionally keep the same public URL and response shape. Their method signatures differ because they are teaching different integration styles.

API Version

@GetMapping("/query")
public ResponseEntity<?> queryWithRegisteredClient(
    @RegisteredOAuth2AuthorizedClient("entra")
    OAuth2AuthorizedClient authorizedClient) {
  String endUserToken = authorizedClient.getAccessToken().getTokenValue();
  List<String> rows = deepDataSecurityService.queryAsEndUser(endUserToken);
  return ResponseEntity.ok(new QueryResponse(rows));
}

The controller receives OAuth2AuthorizedClient because app code explicitly passes the signed-in user's token into Oracle JDBC's EndUserSecurityContext API.

Provider/SPI Version

@GetMapping("/query")
public ResponseEntity<?> query() {
  List<String> rows = deepDataSecurityService.queryAsEndUser();
  return ResponseEntity.ok(new QueryResponse(rows));
}

The provider controller stays clean. Spring Security validates the bearer token before the method runs, and ojdbc-provider-spring reads the authenticated context through JDBC provider SPI when the service borrows a connection and executes SQL.

Oracle UCP Instead of HikariCP

Both demo apps now use Oracle UCP. HikariCP is excluded from the spring-boot-starter-jdbc dependency so the comparison focuses on the Deep Data Security integration style, not on different pool implementations. 

<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-jdbc</artifactId>
  <exclusions>
    <exclusion>
      <groupId>com.zaxxer</groupId>
      <artifactId>HikariCP</artifactId>
    </exclusion>
  </exclusions>
</dependency>

<dependency>
  <groupId>com.oracle.database.jdbc</groupId>
  <artifactId>ucp</artifactId>
  <version>${oracle.jdbc.version}</version>
</dependency>

The API version creates a UCP PoolDataSource, sets the Oracle datasource factory, and uses the normal application database user: 

PoolDataSource dataSource = PoolDataSourceFactory.getPoolDataSource();
dataSource.setConnectionFactoryClassName("oracle.jdbc.pool.OracleDataSource");
dataSource.setURL(required(properties.getJdbcUrl(), "DEEPSEC_JDBC_URL"));
dataSource.setUser(required(properties.getUsername(), "DEEPSEC_USERNAME"));
dataSource.setPassword(required(properties.getPassword(), "DEEPSEC_PASSWORD"));
dataSource.setConnectionPoolName("DeepDataSecurityApiUcpPool");
dataSource.setInitialPoolSize(1);
dataSource.setMinPoolSize(1);
dataSource.setMaxPoolSize(10);

The provider version also creates a UCP PoolDataSource, then adds the JDBC provider settings as UCP connection properties: 

dataSource.setConnectionFactoryProperty(
    "oracle.jdbc.provider.endUserSecurityContext",
    "ojdbc-provider-spring-end-user-security-context");
dataSource.setConnectionFactoryProperty(
    "oracle.jdbc.provider.endUserSecurityContext.registrationId",
    "entra");

The matching YAML makes the pool sizing and provider registration explicit: 

deepsec:
  ucp:
    connection-pool-name: ${DEEPSEC_UCP_POOL_NAME:DeepDataSecurityProviderUcpPool}
    initial-pool-size: ${DEEPSEC_UCP_INITIAL_POOL_SIZE:1}
    min-pool-size: ${DEEPSEC_UCP_MIN_POOL_SIZE:1}
    max-pool-size: ${DEEPSEC_POOL_SIZE:4}
  jdbc-provider:
    end-user-security-context: ojdbc-provider-spring-end-user-security-context
    registration-id: entra

Database and Microsoft Entra Setup

Oracle AI Database must trust the Microsoft Entra tenant and the database/API app registration before it can validate the database-access token. The database/API app registration defines app roles such as EMPLOYEES and MANAGERS. Users or groups are assigned to those roles on the Enterprise Application for that resource. 

CREATE OR REPLACE DATA ROLE hrapp_employees
  MAPPED TO 'AZURE_ROLE=EMPLOYEES';

CREATE OR REPLACE DATA ROLE hrapp_managers
  MAPPED TO 'AZURE_ROLE=MANAGERS';

The application connection-pool user still needs only the privileges required to connect and attach an end-user security context: 

GRANT CREATE SESSION TO hr_app_user;
GRANT CREATE END USER SECURITY CONTEXT TO hr_app_user;

Data Grants

The policy itself lives in SQL. For an employee, the row predicate says a user can see rows where the table username matches ORA_END_USER_CONTEXT.username: 

CREATE OR REPLACE DATA GRANT hr.HRAPP_EMPLOYEES_ACCESS
  AS SELECT, UPDATE(phone_number, first_name)
  ON hr.employees
  WHERE upper(user_name) = upper(ORA_END_USER_CONTEXT.username)
  TO hrapp_employees;

For managers, the policy permits access to direct reports while excluding sensitive columns such as ssn: 

CREATE OR REPLACE DATA GRANT hr.HRAPP_MANAGER_ACCESS
  AS SELECT (ALL COLUMNS EXCEPT ssn),
     UPDATE (salary, department_id, first_name)
  ON hr.employees
  WHERE manager_id = ORA_END_USER_CONTEXT.HR.EMP_CTX.ID
  TO hrapp_managers;

The Java service or AI agent is not building custom SQL for employees and managers. The app can issue the same business query, and Oracle filters or restricts the result according to the active end-user security context.

Expected Demo Result

Run the same endpoint with two role contexts. With an employee token, the same query returns the single permitted employee row: 

{"rows":["3:Emma Baker"]}

With a manager token, the same endpoint and SQL return the manager's permitted direct reports: 

{"rows":["3:Emma Baker","4:Charlie Davis","5:Dana Lee"]}

How To Explain the Difference

Use this short version when presenting:

The API version shows the explicit mechanics: receive the OAuth2 authorized client, get the user's token, request a database token, set the Oracle JDBC end-user security context, run SQL, and clear the context. The provider version keeps the same public endpoint but moves that context setup into the JDBC provider/Spring integration. Both use UCP, both use the same database policy model, and both let Oracle AI Database enforce the authorization.

Production Notes

  • Store Entra client secrets and database credentials in a secrets manager.
  • Validate issuer, audience, scopes, and expected role claims for user tokens.
  • Keep JDBC and UCP artifacts on the same Oracle release version.
  • Manage data grants as versioned database migration scripts.
  • Add tests that run the same SQL as different users or roles and assert row, column, and cell-level behavior.
  • Audit data grants and active end-user contexts with Oracle Deep Data Security dictionary views.
