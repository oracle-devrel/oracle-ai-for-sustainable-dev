# Oracle AI Database Deep Data Security Spring Boot Demo

## TL;DR

First configure Oracle AI Database 26ai for Deep Data Security:

```sql
-- Run as DBA on Oracle AI Database, or ADMIN on Autonomous AI Database.
-- TLS/TCPS must be configured before the app propagates end-user security context.

CREATE USER hr_app_user IDENTIFIED BY "your_database_password";
GRANT CREATE SESSION TO hr_app_user;
GRANT CREATE END USER SECURITY CONTEXT TO hr_app_user;

CREATE OR REPLACE DATA ROLE employee_role;

CREATE OR REPLACE DATA GRANT hr.employees_full_grant
  AS SELECT
  ON hr.employees
  TO employee_role;

-- Also configure the database identity provider trust for Microsoft Entra ID
-- or OCI IAM, then map IAM application roles/groups to database data roles.
-- See the Oracle Deep Data Security Guide links in References.
```

Then configure and run the Spring Boot app:

```bash
cd security/deepdatasecurity
vi .env

./build.sh
./run.sh

# Windows PowerShell:
./build.ps1
./run.ps1

curl -H "Authorization: Bearer END_USER_ACCESS_TOKEN" \
  http://localhost:8080/deepsec/query
```

For a smoke check that does not touch the database:

```bash
curl http://localhost:8080/deepsec/health
```

Replace `DEEPSEC_DATABASE_ACCESS_TOKEN` with a real Microsoft Entra ID or OCI IAM token exchange before using this beyond a local demo.

## What This Demo Shows

Oracle Deep Data Security is available with Oracle AI Database 26ai. It moves fine-grained end-user authorization into the database, so SQL generated by applications, agents, or AI-assisted workflows is evaluated against the effective end user's identity and runtime context instead of only the shared application database account.

This Spring Boot app demonstrates the direct Oracle JDBC API path documented in Oracle's Deep Data Security guide. The same guide also describes a configuration-driven SPI/Spring provider path; this demo intentionally uses the API extension method path so the set/execute/clear lifecycle is visible in application code.

1. The client calls `GET /deepsec/query` with `Authorization: Bearer <end-user-token>`.
2. The app obtains a database-access token. The demo reads it from `DEEPSEC_DATABASE_ACCESS_TOKEN`.
3. The app creates an `oracle.jdbc.EndUserSecurityContext`.
4. The app unwraps the JDBC connection to `oracle.jdbc.OracleConnection`.
5. The app calls `setEndUserSecurityContext(...)`.
6. The app executes the configured SQL.
7. The app calls `clearEndUserSecurityContext()` in a `finally` block before the connection can be reused.

## Why This Matters

Agentic AI applications commonly use a service account or pooled database user while acting on behalf of many users. If the application or agent constructs unexpected SQL, application-layer checks may be bypassed. Deep Data Security keeps the authorization boundary in the database:

- end-user and application identity are propagated to the database at runtime
- declarative SQL policies enforce row, column, and cell-level access
- audit records can reflect the real user or agent, not only the connection-pool user
- application code no longer needs to duplicate every data authorization rule

## Maven Dependency

The app uses the Oracle JDBC driver family shown in Oracle's Deep Data Security Java configuration examples.

```xml
<dependency>
  <groupId>com.oracle.database.jdbc</groupId>
  <artifactId>ojdbc11</artifactId>
  <version>23.26.1.0.0</version>
</dependency>
```

If your deployment needs wallet helper artifacts, add Oracle's `ojdbc11-production` dependency set or the specific wallet/security artifacts that match your JDBC version.

The app uses Oracle Universal Connection Pool (UCP):

```xml
<dependency>
  <groupId>com.oracle.database.jdbc</groupId>
  <artifactId>ucp</artifactId>
  <version>23.26.1.0.0</version>
</dependency>
```

`UcpDataSourceConfiguration` creates a `PoolDataSource` with `oracle.jdbc.pool.OracleDataSource` as the connection factory.

## Database Setup

Yes, the database must be configured before this app can enforce Deep Data Security. At minimum:

- configure TLS/TCPS between the app and database
- create or choose the connection-pool database user
- grant that user `CREATE SESSION`
- grant that user `CREATE END USER SECURITY CONTEXT`
- configure database trust for the IAM provider that issues the database-access token
- create data roles that match IAM application roles or app-supplied data roles
- create data grants or policies for the target tables/views

Without `CREATE END USER SECURITY CONTEXT`, the database rejects attempts to attach the end-user security context payload.

## Application Configuration

The app reads these environment variables:

```text
DEEPSEC_PORT                   optional HTTP port, default 8080
DEEPSEC_JDBC_URL               JDBC URL for the Oracle AI Database 26ai service
DEEPSEC_USERNAME               application or connection-pool database user
DEEPSEC_PASSWORD               password for the application database user
DEEPSEC_DATABASE_ACCESS_TOKEN  temporary database-access token for the demo path
DEEPSEC_SQL                    optional query, default queries an employees table
DEEPSEC_DATA_ROLES             optional comma-separated data roles
DEEPSEC_ATTRIBUTES_JSON        optional JSON object for end-user context attributes
```

`build.sh`, `run.sh`, `build.ps1`, and `run.ps1` load `.env` automatically before invoking Maven.

Default SQL:

```sql
select employee_id || ':' || first_name || ' ' || last_name
from employees
fetch first 10 rows only
```

Override it when your schema uses different tables:

```bash
export DEEPSEC_SQL="select sys_context('USERENV','SESSION_USER') from dual"
```

## TLS Requirement

Oracle documentation states that the database server must be configured for TLS before the application is configured for Deep Data Security. The client trust store or wallet must trust the database server certificate.

For wallet-based JDBC connections, include the wallet settings in the JDBC URL or JVM properties according to your Oracle JDBC deployment pattern.

## Direct JDBC API Shape

The implementation follows Oracle's documented API extension method flow:

- build an `EndUserSecurityContext` with `createWithToken(...)`
- unwrap the `java.sql.Connection` to `OracleConnection`
- call `setEndUserSecurityContext(...)` before SQL execution
- close `ResultSet`, `Statement`, and `Connection`
- call `clearEndUserSecurityContext()` in `finally` before the connection can be reused

The core Java pattern is:

```java
EndUserSecurityContext context =
    EndUserSecurityContext.createWithToken(databaseAccessToken, endUserToken)
        .withDataRoles(Set.of("employee_role"))
        .withAttributes(Map.of(
            "hr.hcm_context",
            Map.of("emp_id", 3, "service_center_id", 5)));

OracleConnection oracleConnection = connection.unwrap(OracleConnection.class);
oracleConnection.setEndUserSecurityContext(context);
try {
    // Execute SQL as the effective end user.
}
finally {
    oracleConnection.clearEndUserSecurityContext();
}
```

Use `EndUserSecurityContext.createWithUsername(databaseAccessToken, endUserName, key)` instead of `createWithToken(...)` when end users are managed locally rather than through IAM tokens.

## Replacing the Demo Token Service

`EnvironmentDatabaseAccessTokenService` is intentionally simple. It lets you run the app once you already have a database-access token.

For production-style demos, replace it with a service that requests a database-access token from IAM:

- client credentials flow when the application acts as itself
- on-behalf-of flow when the database-access token should be minted from the end-user assertion

Keep token caching in that replacement service. Database-access tokens are short-lived, and OBO tokens should generally be cached per end-user token or per user/session key.

## SPI Option for Frameworks

For larger framework applications, the JDBC Service Provider Interface can reduce direct application changes. Implement or install an `oracle.jdbc.spi.EndUserSecurityContextProvider`; the driver calls it before database operations and attaches the returned `EndUserSecurityContext` payload. Oracle's Spring Boot provider can connect this to Spring Security OAuth 2.0 configuration.

Oracle's Spring provider documentation configures:

```yaml
oracle.jdbc.provider.endUserSecurityContext: ojdbc-provider-spring-end-user-security-context
oracle.jdbc.provider.endUserSecurityContext.registrationId: hrapp
```

With UCP, pass provider properties to the Oracle connection factory or set them as connection properties on the pool. That provider-based mode is best when Spring Security owns authentication and you want the JDBC driver to attach the security context automatically. The code in this demo uses the API path instead so the mechanics are easy to see and adapt.
